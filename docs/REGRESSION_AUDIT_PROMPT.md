# ai-fc 내부 회귀·고도화 감사 프롬프트 — REGRESSION_AUDIT v1.0

> **용도**: 작업폴더(리포) 내부에서 "구조화·고도화할 것이 남아 있는가"를 **증거 기반으로** 검증하는 정기 감사.
> **주기**: 월 1회 (첫 토요일 주간 리프레시 직후) + 체크포인트(8/31·10/31·12/31) + 대형 변경 직후.
> **기존 SELF_AUDIT_PROMPT와의 관계**: 그것은 헌법 준수 감사, 이것은 **회귀(깨진 것) + 정합(어긋난 것) + 발굴(고도화 후보)** — 상호 보완.
> **보관**: 결과는 `reports/audit/_local_only/regression_YYMMDD.md` (로컬 전용 — 공개 리포 미포함).

## 실행 프롬프트 (아래 블록 전체를 Claude Code에 붙여넣기)

```
ai-fc 리포에서 내부 회귀·고도화 감사를 수행하라 (docs/REGRESSION_AUDIT_PROMPT.md v1.0).
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[역할] 너는 이 시스템을 처음 보는 외부 감사관의 시선으로 내부를 검증한다.
       애착 없이, 그러나 파괴 충동도 없이.
[제0원칙 — 발견을 만들어내지 마라]
- "이상 없음"은 실패가 아니라 최선의 결과 후보다. 보고서를 채우기 위한 지적 금지.
- 모든 발견(Finding)은 증거 필수: 파일:줄 번호, 명령 실행 출력, 또는 수치 대조.
  증거를 못 대면 발견이 아니라 "인상"이며, 인상은 보고서에서 제외한다.
- 고도화 후보는 §G의 5필드 양식을 전부 채워야만 후보 자격을 갖는다. 상상에서
  후보를 만들지 마라 — A~F 검사에서 실측된 마찰·회귀·병목에서만 도출한다.
[실행 제약]
- 읽기 전용. 허용되는 쓰기는 보고서 파일 1개뿐. 종료 시 git status로
  "보고서 외 변경 0"을 스스로 증명하라 (§I 자가 검증).
- LLM API 호출 금지 (비용 $0) — 로컬 실행·정적 분석·기존 산출물 대조만.
- 봉인 목록(하단)을 존중하라. 봉인 항목을 후보로 올리려면 "봉인 해제 사유"
  필드를 추가로 채워야 하며, 그 판단은 어차피 사용자 몫이다.
- 보고서 제출 후 정지. 수정 실행은 승인 후 별도 세션이다.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
다음 8개 섹션을 순서대로 검사하고, 각 섹션에 판정(정상/경미/조치필요)을 내려라.
── A. 불변성·원장 회귀 (P0 후보 영역) ──────────────────────
A1. pytest 전체 실행 — 통과/실패/스킵 수를 직전 기록과 대조. 실패 1건이라도 있으면
    원인 파일:줄과 "언제부터"(git log)를 특정하라. 테스트 수정으로 통과시키는 것 금지.
A2. python tools/verify_track_record.py 실행 — 등급 카운트(A/B)와 PASS 여부.
    직전 실행 대비 카운트 변화가 있다면 전부 커밋으로 설명 가능한가.
A3. sync --check exit code. E1~E7 각각의 검출 대상이 코드에 실재하는지 grep 확인.
A4. forecasts/ 파일 수 == .hashes 항목 수 == DB 예측 행 수 — 삼자 정합.
A5. git status 클린 여부, 추적 안 된 파일 목록 (있다면 각각 정체 규명).
── B. 문서↔코드 정합 (거짓말 탐지) ─────────────────────────
B1. README의 모든 정량 주장(테스트 수·검사 종수·단가·게이트 수치·질문 수)을
    코드·설정·실측에서 하나씩 재확인 — 문서가 코드보다 좋게 말하는 지점 = 발견.
B2. CLAUDE.md 원칙 각각에 대해 "그 원칙을 집행하는 코드 지점" 1개 이상을 파일:줄로
    매핑하라. 매핑 불가 원칙 = 선언만 있고 집행 없음 = 발견.
B3. DECISIONS 최근 10건 — 각 결정이 코드에 실재하는지 (뒤집혔는데 기록 안 된 것 포함).
B4. KNOWN_LIMITS 전 항목 — ① 여전히 사실인가 ② 해소됐는데 남아있나(stale)
    ③ 새로 생겼는데 없는 한계가 있나. 셋 다 발견 대상.
B5. MODEL_REGISTRY 상태 vs 코드 — demoted 모델이 실제로 비활성인가(LPPL 게이트 등),
    승격 없이 사용되는 모델이 있나.
── C. 스펙 DoD 회귀 ────────────────────────────────────────
C1. v2·v3·v3.5 스펙의 DoD 항목 전수(유한하다)를 현재 시점에 재검 —
    "완료였는데 지금은 깨진 것"이 회귀다. 각 DoD에 [유지/회귀/N/A] 판정.
── D. 코드 구조 위생 (P2 영역) ─────────────────────────────
D1. TODO/FIXME/HACK/XXX 전수 목록 — 각각 [유효/폐기가능/승격필요] 분류.
D2. 어디서도 import되지 않는 모듈·함수 (정적 스캔) — 단, 의도적 이중구현
    화이트리스트(검증기의 해시·원장 로직 = sync와 독립이 설계 의도) 제외.
D3. config.py 밖의 하드코딩 상수 스캔 — 임계값·경로·모델명이 코드에 박힌 지점.
D4. 모듈별 대응 테스트 존재 매트릭스 — 테스트 0개 모듈 목록 (커버리지 도구 없이
    파일 대응으로만, 과장 금지).
D5. pyproject 의존성 선언 vs 실제 import 대조 — 미선언 사용·미사용 선언.
── E. 데이터 계층 위생 ─────────────────────────────────────
E1. base_rates 11종 각각의 asof·신선도(7일 규칙) — stale 목록.
E2. dualdb 커버리지: 핵심 시리즈 결측률이 직전 리포트 대비 악화됐는가.
E3. 센티널 전 항목 재실행 결과.
E4. 이중 소스 불일치 현황 — 미결 결정(예: Yahoo↔FRED 2003~04 건)의 상태와
    방치 기간. 결정 대기가 30일을 넘기면 P1로 승격 보고.
E5. ml_history·원장류 행 수의 단조 증가 확인 (감소 = P0).
── F. 운영 정합 (계획 vs 실행) ─────────────────────────────
F1. HARVEST_CALENDAR vs registry — 캘린더의 모든 r1 마감·해소 예정이 registry
    질문과 1:1 대응하는가. 고아 항목(캘린더에만/레지스트리에만) 목록.
F2. 지난 주차들의 캘린더 이행 실적 — 계획된 r1/재예측/해소/회고가 실제 수행됐는가.
    미이행 = 발견 (사유 불문 기록, 판단은 사용자).
F3. 비용: 로그 합계 vs 월 상한 vs 캘린더 추정 — 편차 ±30% 초과 시 발견.
F4. 예약 태스크 생존 확인 — 최근 실행 흔적(로그·산출물 mtime).
F5. 해소 건수 == retro 파일 수 == benchmark_ledger 신규 행 수 — 리추얼 준수.
── G. 고도화 후보 발굴 (P3 — 엄격 양식) ────────────────────
A~F에서 실측된 것에서만 도출. 후보당 5필드 전부 필수:
  ① 푸는 병목: (표본 속도 / 정확성 / 신뢰·검증 / 운영 마찰) 중 무엇이며 근거 수치
  ② 증거: 이번 감사의 어느 Finding에서 나왔나 (섹션·번호)
  ③ 헌법·봉인 저촉 검토: 통과/저촉 (저촉이면 봉인 해제 사유 별도 서술)
  ④ 예상 비용: 구현 시간 + 운영비 변화
  ⑤ 하지 않을 경우: 무엇이 나빠지나 (아무것도 안 나빠지면 후보 탈락)
후보 0건이면 "0건 — 시스템이 현 목적에 충분"이라고 명시하라. 그것이 정답일 수 있다.
── H. 리스크 레지스터 갱신 제안 ────────────────────────────
KNOWN_LIMITS에 추가/해소/수정할 항목 초안 (B4 결과의 실행 형태).
외부 환경 변화(데이터 소스 접근성·모델 버전·단가) 중 문서 미반영분.
── I. 자가 검증 후 보고서 제출 ─────────────────────────────
git status 출력 첨부 — 보고서 1파일 외 변경 0 증명.
━━━━━━━━━━━━━ 보고서 형식 (고정 템플릿) ━━━━━━━━━━━━━
# 회귀·고도화 감사 — YYYY-MM-DD (커밋 <hash>)
## 최종 판정: [정상 — 조치 불요 | 경미 — P2 이하 N건 | 조치 필요 — P0/P1 존재]
## 심각도 집계: P0 _건 · P1 _건 · P2 _건 · P3(후보) _건
   (P0 불변성·정합 파괴=즉시 / P1 자인 규율 위반·회귀·30일+ 방치 결정 /
```

## 운영 규칙

| 항목 | 규칙 |
|---|---|
| 주기 | 월 1회 정기 + 체크포인트 3회(8/31·10/31·12/31) + 대형 변경 직후 임시 |
| 첫 실행 | "기준선 수립" 회차 — 직전 대비 항목은 N/A, A1·A2의 대조 기준값을 이 보고서가 확정 |
| 결과 처리 | P0 = 즉시 사용자 보고(감사 세션 내 수정 금지) · P1 = 차기 작업 세션 1순위 · P2 = 배치 처리(분기 1회 묶음) · P3 = 사용자 승인 시에만 별도 라운드 |
| 프롬프트 자체의 개정 | 이 문서도 버전 관리 — 감사가 "검사 항목 추가/삭제"를 제안할 수 있으나(§H) 개정은 사용자 승인 + 갱신 이력 기록. 감사가 자기 기준을 스스로 완화하는 것 금지 |
| 비용 | $0 (LLM API 무호출) — 감사는 언제나 공짜여야 미루지 않는다 |
| 시간 상한 | 1회 60~90분 분량. 초과 시 남은 섹션을 "미검"으로 정직하게 표기하고 차회 이월 |
